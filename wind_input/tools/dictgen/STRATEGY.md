# dictgen 简码降权策略

## 背景

五笔86输入法中，一个字可能同时拥有简码（2码或3码）和全码（4码）。当该字在简码和4码全码中都占据首选位置时，4码的候选位被浪费——用户已经可以通过简码打出该字，4码本应展示其他有价值的候选。

**示例**：`阪` 是二级简码 `br` 的首选，同时又是 `brcy` 的首选。用户输入 `brcy` 时，如果想打 `阪` 直接用 `br` 即可，`brcy` 的首选位应该让给更有价值的候选。

## 策略规则

### 规则 1：过滤 — 忽略极低频候选

**条件**：第二候选权重 < `filter_threshold`（默认 200）

**动作**：不参与降权计算，保持原顺序

**理由**：极低频词/生僻字不值得占据首位，如 `百公里(150)`、`戜(150)`、`莕(120)`

### 规则 2：降权 — 简码字权重降到第二候选之下

**条件**（同时满足）：
- 单字第二候选：`2nd_is_char=Y` 且 `2nd_wt >= single_char_promote_wt`（默认 1000）且 `gap/首字wt < max_gap_ratio_single`（默认 0.60）
- 词组第二候选：`2nd_is_char=N` 且 `2nd_wt >= word_promote_wt`（默认 800）且 `gap/首字wt < max_gap_ratio_word`（默认 0.65）

**动作**：将简码字的权重降到 `第二候选权重 - 1`

**理由**：第二候选是有价值的常用字/词，且权重差距不足以让首字"理所当然"占据首位

**测试验证案例**：
| 首字(wt) | 第二候选(wt) | 类型 | gap | gap/首字wt | 结果 |
|---|---|---|---|---|---|
| 基(1833) | 斟(1365) | 单字 | 468 | 0.25 | 降权 |
| 孤(1732) | 阪(1148) | 单字 | 584 | 0.34 | 降权 |
| 帮(1915) | 邦(1628) | 单字 | 287 | 0.15 | 降权 |
| 难(2006) | 验证(1398) | 词 | 608 | 0.30 | 降权 |
| 原(1984) | 矿泉水(1200) | 词 | 784 | 0.40 | 降权 |
| 现(1950) | 一见钟情(1141) | 词 | 809 | 0.41 | 降权 |
| 茎(1271) | 芭蕾(1114) | 词 | 157 | 0.12 | 降权 |
| 靶(1305) | 鞍马(922) | 词 | 383 | 0.29 | 降权 |

### 规则 3：保留 — 维持原顺序

**条件**：不满足规则 1 和规则 2

**典型场景**：
- 首字非常高频（>2200）且 gap 大：如 `大(2262) vs 硕大(994)`，gap/首字wt=0.56
- 第二候选权重不够高：如 `苛(1323) vs 莕(120)`，过滤阈值不满足
- 两个都是生僻字：如 `葚(848) vs 慝(619)`，都不够常用

**测试验证案例**：
| 首字(wt) | 第二候选(wt) | 类型 | gap | gap/首字wt | 结果 |
|---|---|---|---|---|---|
| 或(2236) | 戜(150) | 单字 | 2086 | 0.93 | 保留 |
| 苛(1323) | 莕(120) | 单字 | 1203 | 0.91 | 保留 |
| 大(2262) | 硕大(994) | 词 | 1268 | 0.56 | 保留 |
| 葚(848) | 慝(619) | 单字 | 229 | 0.27 | 保留 |
| 菲(1737) | 匪(1564) | 单字 | 173 | 0.10 | 保留 |
| 若(1899) | 苦味(1041) | 词 | 858 | 0.45 | 保留 |

## 配置参数

在 `dictgen.yaml` 的 `demotion` 段配置：

```yaml
demotion:
  enabled: true                # 是否启用降权策略
  filter_threshold: 200        # 第二候选权重低于此值直接忽略
  single_char_promote_wt: 1000 # 单字第二候选权重 >= 此值时触发降权
  word_promote_wt: 800         # 词组第二候选权重 >= 此值时触发降权
  max_gap_ratio_single: 0.60   # 单字：gap/首字权重 > 此比例则保留
  max_gap_ratio_word: 0.65     # 词组：gap/首字权重 > 此比例则保留
```

### 参数调优指南

- **filter_threshold**：调高会忽略更多低频候选（更保守），调低会处理更多条目（更激进）
- **single_char_promote_wt / word_promote_wt**：调高则只有更高频的候选才能触发降权（更保守）
- **max_gap_ratio_single / max_gap_ratio_word**：调高则允许更大的权重差距仍进行降权（更激进）

### 调优流程

1. 修改 `dictgen.yaml` 中的参数
2. 运行 `dictgen` 生成词库
3. 检查 `demotion_report_path` 输出的报告，确认降权效果
4. 在实际输入法中测试候选排序是否符合预期
5. 微调参数并重复

## 实现位置

### 词典层（tools/dictgen/）

- 配置：`config.go` — `DemotionConfig` 结构体
- 逻辑：`enrich.go` — `applyDemotionStrategy()` 函数
- 调用：`enrich()` 函数中，权重计算之后、排序之前
- 降权目标：只修改4码条目，不动简码条目

### 引擎层（internal/engine/codetable/）

问题：同一字在不同码长有不同权重时，引擎合并候选时若把简码条目放在4码条目之前，
去重（`dedup`）按"首次出现"保留就会让简码条目覆盖降权后的4码条目。

修复：保证 `exactCandidates` 始终在 `prefixCandidates` 之前合入：
```go
allCandidates := append(exactCandidates, prefixCandidates...)
allCandidates = dedup(allCandidates)
```
当输入码长=4 时，4码条目以"精确匹配"身份先入，简码条目以"前缀匹配"身份后入并被去重跳过，
4码降权效果得以保留。`dedup` 自身保持"首入者优先"的简单语义，无需额外标记字段。
